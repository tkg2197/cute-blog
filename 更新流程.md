# 「我们的小窝」更新流程

记录改代码 → 本地测试 → 部署上线的完整流程。以后想加功能/改样式照着这个走就好。

## 项目结构速览

```
cute-blog/                      ← GitHub 仓库根 (本目录就是)
├── astro.config.mjs            ← Astro 配置（含 Vercel adapter、sin1 区域）
├── vercel.json                 ← Vercel 部署配置
├── package.json                ← 依赖清单
├── public/                     ← 静态资源（图片、字体、独立脚本/样式）
├── src/                        ← Astro 源代码
│   ├── pages/                  ← 路由页面
│   ├── layouts/                ← 布局组件
│   ├── lib/                    ← 工具函数 / Supabase 客户端
│   └── middleware.ts           ← 全局中间件 (auth)
├── supabase/migrations/        ← 数据库迁移 SQL
└── .env                        ← 本地环境变量（已 gitignore，不入仓）
```

部署架构：

```
你的浏览器
  ↓
Vercel (sin1 新加坡, SSR)
  ↓
Supabase (新加坡, 数据库 + 存储 + Auth)
```

## 一、四种测试方式（按从简到全排）

### 方式 1：本地热更新开发 ⚡ 最常用

改代码立刻看到效果，秒级反馈。

```bash
cd astro
npm run dev
```

访问 http://localhost:4321 。改任何 `.astro` / `.css` / `.js` 浏览器自动刷新。

- 改 CSS：热替换，不刷新就生效
- 改 .astro / .ts：整页刷新
- 按 `Ctrl+C` 退出

**适用**：调样式、改文案、改交互、加新页面。

---

### 方式 2：本地生产构建预览 🔍 推送前的关卡

dev 模式跟生产环境有一些差异（dev 不压缩、有 source map）。重要修改 push 前先跑一次生产构建确认不报错。

```bash
cd astro
npm run build
```

看输出。**有红色报错 → 不要 push，先修**。无错就放心。

想本地跑跑生产版本看看：

```bash
npm run preview
```

访问 http://localhost:4321 ——这次是压缩、SSR 的真生产版。

---

### 方式 3：Vercel Preview Deployments 🌿 不动主域名的线上实验

每个**非 main 分支**推到 GitHub 后，Vercel 自动构建一个独立 URL，跟生产环境一模一样（同 region、同环境变量、同 SSR），但**不影响主域名**。

```bash
cd astro

# 1. 从 main 切新分支
git checkout -b try-new-feature

# 2. 改代码 ...

# 3. 提交 + 推送
git add .
git commit -m "尝试新功能"
git push -u origin try-new-feature
```

Vercel 自动构建，几分钟后给你一个 preview URL：
```
https://cute-blog-git-try-new-feature-tkg-s-projects.vercel.app
```

打开试一下：

- **满意** → 合并到 main 部署生产：
  ```bash
  git checkout main
  git merge try-new-feature
  git push
  # 顺手删分支
  git branch -d try-new-feature
  git push origin --delete try-new-feature
  ```
- **不满意** → 直接删分支，主域名完全没动过：
  ```bash
  git checkout main
  git branch -D try-new-feature
  git push origin --delete try-new-feature
  ```

---

### 方式 4：GitHub Pull Request 👀 两个人一起决定的功能

跟方式 3 同原理，但走 PR 让两个人都能讨论。

1. 分支推上 GitHub 后，仓库页面会有 **Compare & pull request** 按钮
2. 点开 → 写一下"改了什么 / 为什么"→ Create pull request
3. PR 页面里会自动出现 Vercel preview URL 链接（"Visit Preview"）
4. 两个人都可以在 PR 里留评论
5. 拍板后点 **Merge pull request** → 主分支自动部署生产

## 二、推荐的日常工作流（按改动风险大小）

| 改动类型 | 走哪个流程 |
|---|---|
| 改文案、调颜色字号、微调样式 | **方式 1（本地 dev）→ 直接 push main** |
| 加新页面、改 API、改组件结构 | **方式 1 → 方式 2 build 验证 → push main** |
| 大功能（新模块、改 Supabase 表） | **方式 1 → 方式 2 → 方式 3 分支预览 → 合并** |
| 两个人共同决定的功能 | **方式 1 → 方式 2 → 方式 3 → 方式 4 PR 讨论 → 合并** |

## 三、Supabase 迁移（改数据库结构时）

数据库表 / RLS 策略 / 触发器之类的修改，**必须先在本地写好 SQL → 在 Supabase 上手动跑一次 → 再写代码使用新表**。

流程：

1. 在 `supabase/migrations/` 新建一个文件，命名 `00X_<功能名>.sql`（X 递增）
2. 写 SQL（参考已有的 4 个 migration 文件）
3. 打开 https://supabase.com/dashboard → 你的项目 → **SQL Editor**
4. 粘贴 SQL 内容 → **Run**
5. 看到 "Success. No rows returned" 即可
6. 再回来写应用代码使用这个表

> ⚠️ 不要直接在 Supabase Dashboard 改表（点 GUI 那种），改完文件不入仓库就丢了。所有变更走 SQL 文件。

### 已有的 4 个迁移

| 文件 | 内容 |
|---|---|
| `001_initial_schema.sql` | profiles + blog_posts + photos + storage buckets + RLS |
| `002_life_records.sql` | life_records 表 |
| `003_activity_entries.sql` | activity_entries 表 |
| `004_places.sql` | places 表（今年想去） |

## 四、环境变量（敏感配置）

### 本地 `.env`（已 gitignore，不会上传）

```text
SUPABASE_URL=https://<your-project>.supabase.co
SUPABASE_ANON_KEY=<publishable / anon key>
SUPABASE_SERVICE_ROLE_KEY=<secret service-role key>
# 可选：限定哪些邮箱能注册
# ALLOWED_SIGNUP_EMAILS=white@example.com,brown@example.com
```

### Vercel 上的环境变量

在 Vercel Dashboard → Project Settings → **Environment Variables** 维护。三个 KEY 必须有，Production / Preview / Development 三个环境都要勾上。

**改了 Vercel 环境变量后，不会自动重部署**——需要去 Deployments 标签手动 redeploy 一次。

## 五、一些容易踩的坑

### ⚠️ 坑 1：本地开发的是真生产数据库

`astro/.env` 里的 SUPABASE_URL 指向**生产 Supabase 项目**。意思是：

- ✅ 本地登录看到的账号 / 数据都是真的
- ❌ 本地不小心删一条记录，**生产环境也删了**
- ❌ 本地试新 SQL 迁移，**会改到生产库结构**

如果以后想要 100% 安全实验，可以建第二个 Supabase 项目作为 dev 环境：
1. Supabase 新建项目
2. 跑同样的迁移
3. 本地 `.env` 换成 dev 项目的 URL/KEY
4. Vercel 上保留生产项目的配置不变

### ⚠️ 坑 2：忘了 push 第二个 commit

本地多个 commit 没全 push 就部署，Vercel 会用 GitHub 上有的那个 commit 部署。每次改完都跑：

```bash
git log --oneline -5            # 看本地有几个 commit
git push                        # 全部推上去
```

### ⚠️ 坑 3：改了 .env 又 push

如果不小心把 `.env` 加进 git，**Supabase service role key 就泄漏了**。万一发生：

1. 立刻去 Supabase Dashboard 重置所有 key
2. 在 Vercel 更新新 key
3. 把仓库里的 `.env` 删掉 + `git push --force`（或干脆 rewrite history）

我们已经在 `.gitignore` 里挡了 `.env`，正常情况不会出问题。但**别手动 `git add .env`**。

### ⚠️ 坑 4：vercel.json 改 region

`vercel.json` 里的 `regions: ["sin1"]` 不要乱改。如果改了，Vercel 函数和 Supabase 之间会重新隔出几百毫秒延迟。

## 七、出问题怎么排查

### 部署失败（Vercel 红色 ✗）

1. 进 Vercel Dashboard → 失败的 Deployment
2. 看 **Build Logs**，红色那行就是错
3. 常见原因：
   - **import 路径错** → 改正
   - **缺包** → 本地 `npm install <包>` → push
   - **类型错误** → 改 ts/astro 文件
   - **环境变量缺** → Vercel Settings 加上

### 部署成功但 404

- 检查 Build Logs 末尾的 `adapter: @astrojs/vercel`，**必须**是 vercel 不是 node
- 检查 `Functions` 标签有没有列出函数（应该有几个）
- 如果都没有问题，多半是某个页面运行时报错，看 **Runtime Logs**

### 网站打开很慢

- 看 Network 标签的 TTFB（首字节）
- 正常应该 300-600ms
- 如果 > 1500ms：可能 Vercel 区域错了 / Supabase 区域错了 / 函数冷启动
- 看 Vercel **Functions** 标签确认 region 是 `sin1`

### 数据库报错 "table xxx does not exist"

新加了 migration 文件但没在 Supabase 上跑过。去 SQL Editor 跑一下。

## 八、流程小结（每次改东西的最简流程）

最常见的"改个文案/调个颜色"场景：

```bash
# 1. 起本地服务器
cd C:\Users\Jason\cute-blog\astro
npm run dev

# 2. 改代码 → 浏览器看效果 → 满意了

# 3. Ctrl+C 退本地服务器

# 4. 推送
git add .
git commit -m "改了什么"
git push

# 5. 等 Vercel 自动部署 1-3 分钟
# 6. 打开 https://你的域名.com 看线上效果
```

复杂改动多走"分支预览"：

```bash
git checkout -b 改名字
# 改代码、本地测试...
git add . && git commit -m "..."
git push -u origin 改名字
# 等 Vercel preview → 用 preview URL 测
# 满意了：
git checkout main
git merge 改名字
git push
git branch -d 改名字
git push origin --delete 改名字
```
